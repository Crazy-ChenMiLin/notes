## OA 登录

目前我们组内的项目在接入登录/注册和身份认证，由于依然在内网使用，所以优先支持内网用户采用 OA 验证登录，在tx内部的话，内网的网站域名基本上都是 `*.woa.com`，然后应企业要求，这些域名都需要接入内部的一个叫“太湖”的平台，然后它会给你发一个 Token，用于后台服务解析它携带的用户信息。

这里主要的作用就是，将网站接入企业网关。对于一次内网请求，大概路径是这样的：
![[1280X1280.png]]


它这里的 Header 基本上就包含了当前的企业用户信息，那对于 OA 登录/注册，最简单的做法就是在后台做一个拦截器嘛，于是我最开始就这么做了，大概像这样（图中省略了本地代理）：
![[1280X1280-1.png]]
这里看上去没有什么问题对吧，但其实可以发现，这里 OA 用户登录是后台所需的业务逻辑。因为按道理来说，只要你带上了这些 Header，并解析出了信息，后台就能够判断你是 OA 访问了，在拦截器就给你返回了。

但现在的问题是，我们的后台要明确的知道，你当前登录的用户的来源，并且要存到数据库，

```TypeScript
user_source // 标明用户来源
```

这里就很蛋疼了，我可以在拦截器做这样的一个操作，
#### 但设计上不允许，让拦截器介入业务，基本上不这么做，果断下一个方案


| 角色    | 流程步骤                                                                                              |
| ----- | ------------------------------------------------------------------------------------------------- |
| OA 用户 | 进入系统 → 前端自动调用`OALogin` → 后端解析 HTTP Header 生成特制 Token → 携带 Token 访问所有接口 → 后端验证 Token 通过 → 进入企业管理系统 |
| 普通用户  | 进入系统 → 前端渲染普通登录界面 → 用户手动登录 / 注册 → 后端生成普通 Token → 携带 Token 访问接口 → 验证通过后进入系统                        |

那就只有回到最原始的方法，让前端主调接口，后台提供一个 `OALogin` 的方法，并且只在这个接口里面解析 OA 用户的 HTTP Header，并且，为了更好的区分内网用户和外网用户，我们最后的流程变成了这样：

![[1280X1280 1.png]]

其实可以看到这里改动还是比较大的，同时也给 OA 用户加上了 Token 进行身份验证，因为对我们的系统来说，企业用户也算是一种“用户”，他们也有用户ID、用户名等信息。然后对于前端来说，改动比较大的地方就是，一旦拿到401，就会默认先调用一次 OALogin，因为前端对 OA 用户是无感知的。

  

最终看到的效果就是，当用户进入我们的登录界面：对 OA 用户来讲，会先看到一个普通用户登录界面的“残影”，然后立马进入主页，这也是没有办法的办法；对于普通用户来说看到的就是一个正常的登录界面了

  

这里关键点就在于后端需要把这些信息存起来，需要明确的知道你是来源于 OA。但一般来说其实是可以直接通用的，也就是说你只要是企业用户，你就必然是我们系统的用户，后台也是无感知的，实现起来就像刚开始我说的那样。

## 关于第三方登录（微信&手机号）

  以下两种方式都需要做的操作是：

- 申请微信平台的appID，获取相关的配置参数比如appsecret
    
- 配置微信扫码过后的回调域名
    
- 在用户确认扫码后，获取微信回调携带的code参数，将其传回给后端
    

###   跳转到微信提供的独立扫码页面登录

  这种方法很简单，我们只需要构造固定的url，然后在我们的应用中跳转到这个url，等待用户扫码后，微信会通过redirect参数回到我们的应用入口，同时在应用入口的url中拼接上code参数。我们在此可以获取code并传给后端。

```JavaScript
const appid = "你的公众号APPID";
const redirectUri = encodeURIComponent("https://yourdomain.com/"); 

export function redirectToWeChatAuth() {
  const scope = "snsapi_userinfo"; //用户授权的等级
  const state = "custom_state";  // 微信扫码后会直接返回的state值，可以用于简单鉴定是否为正常微信扫码
  
  //这个url构造模式为微信平台官方指定
  const url = `https://open.weixin.qq.com/connect/oauth2/authorize?appid=${appid}&redirect_uri=${redirectUri}&response_type=code&scope=${scope}&state=${state}#wechat_redirect`;

  window.location.href = url;
 
```

  外部调用 redirectToWeChatAuth 方法后，就会进入微信扫码页面，完成扫码后就会像上面说的一样返回code重定向

  

###   使用微信平台提供的前端脚本定制网页内嵌的微信扫码

  这种方法需要获取微信提供的 wxLogin.js 配合将其封装为需要的样式，随后在应用中直接渲染到对应的位置

  

  首先需要引入，在各个类型的前端项目中，通常都是直接在主要 index.html中直接引入而不走打包流程

  （gpt说的）

```Plain
<script src="https://res.wx.qq.com/connect/zh_CN/htmledition/js/wxLogin.js"></script>
```

  使用react的话就可以类似于下面这种形式

```JavaScript
export default function WechatQrLogin({
  appid,
  redirectUri,
  state = "csrf_" + Math.random().toString(36).slice(2),
}: WechatQrLoginProps) {
  useEffect(() => {
    // 确保 wxLogin.js 已经在 index.html 里引入
    if (!window.WxLogin) return;

    new window.WxLogin({
      self_redirect: false,
      id: "wx_login_container",
      appid,
      scope: "snsapi_login",
      redirect_uri: encodeURIComponent(redirectUri),
      state,
      style: "black",
      href: "",
    });
  }, [appid, redirectUri, state]);

  return <div id="wx_login_container" />;
}
```

  

此外，我们的项目中暂时使用到的，相当于用户唯一凭证的身份信息为手机号，所以我们要求新用户即使选择了微信扫码登录，也需要绑定手机号才能进入系统。
