# 原理：
## 问题一：上传文件识别和 OCR 识别的核心区别

| 对比维度   | 本次识别方案                        | 传统 OCR 识别                                 |
| :----- | :---------------------------- | :---------------------------------------- |
| 核心处理对象 | 电子生成的可编辑 PDF，底层是数字化字符流        | 纸质扫描件、拍照图片、图像版 PDF，底层是文字像素                |
| 核心逻辑   | 直接读取文档底层字符编码，无损提取原文           | 图像预处理 + 文字区域检测 + 像素转字符，算法还原文本             |
| 准确率    | 100% 匹配原文，无识别误差               | 受图片清晰度、字体、排版、光线影响，有概率出现错字、漏识别、音标乱码        |
| 核心适用场景 | 电脑生成的可编辑 PDF/Word/Excel 等电子文档 | 纸质文件拍照、扫描件、图片里的文字提取                       |
| 说人话-例子 | 看着答案写作业（无脑誊抄）                 | 自己写作业（靠脑子想）                               |
| App 应用 | Ai 理解 pdf/excel 文件的内容         | 微信的图片识别功能（轻量级）/全能扫描王OCR 文字提取（重量级）/相机的文档模式 |

# 问题三：为什么不转化为 excel?/excel 文件的本质

| 对比维度       | CSV（豆包可直接生成）             | Excel（.xlsx，豆包不直接生成）                              |
| :--------- | :----------------------- | :------------------------------------------------ |
| **文件本质**   | 纯文本格式，仅字符 + 分隔符（逗号 / 换行） | **OOXML 格式**：本质是 ZIP 压缩包，内含多层 XML 结构              |
| **内部结构**   | 无结构，纯数据流                 | 包含：工作表、样式、字体、颜色、公式、图表、批注、多 Sheet 等完整文档对象          |
| **生成难度**   | 字符串拼接即可，无任何库依赖           | 必须用专业 SDK（POI/Epplus/OpenXML），要构建 XML 树、压缩打包、格式校验 |
| **模型适配性**  | 与大模型**文本生成天然匹配**         | 属于**二进制 / 结构化文档生成**，超出对话式文本模型的原生能力                |
| **安全风险**   | 纯文本，无执行逻辑，零安全风险          | 可嵌入 VBA 宏、OLE 对象，存在代码执行、恶意文件风险                    |
| **IO 与性能** | 极轻量，内存 / CPU 开销可忽略       | 需序列化、压缩、IO 写入，客户端 / 服务端开销都大                       |
| **跨平台**    | 任何系统 / 语言可直接解析           | 必须依赖办公软件的解析引擎，跨平台易出现格式兼容问题                        |
