# 外键约束

```
-- 1. 先建主表：学生表（必须先建主表，再建子表）
CREATE TABLE 学生表 (
    学号 INT PRIMARY KEY AUTO_INCREMENT,
    姓名 VARCHAR(20) NOT NULL
);

-- 2. 再建子表：成绩表，加外键约束
CREATE TABLE 成绩表 (
    成绩ID INT PRIMARY KEY AUTO_INCREMENT,
    学号 INT,  -- 这一列是外键，引用学生表的学号
    科目 VARCHAR(20) NOT NULL,
    分数 INT NOT NULL,
    -- 加外键约束：成绩表的学号，必须引用学生表的学号
    constraint 外键约束名 FOREIGN KEY(学号) REFERENCES 学生表(学号)
);
```
性能影响
	分布式根本不能用
	高并发情况下：查询多个表卡死
	蝴蝶效应：主表改了
	
企业里的通用做法：**外键的逻辑，放到程序代码里做校验**，数据库层面只存这个列，不加外键约束。




Eg:

``` sql
CREATE TABLE 员工表 (
    -- 主键+自增长：员工工号，自动递增，唯一非空
    emp_id INT PRIMARY KEY AUTO_INCREMENT COMMENT '员工工号',
    -- 非空约束：姓名必须填
    emp_name VARCHAR(20) NOT NULL COMMENT '员工姓名',
    -- 唯一约束：手机号不能重复
    emp_phone CHAR(11) UNIQUE COMMENT '员工手机号',
    -- 默认值约束：不填性别，默认保密
    emp_gender CHAR(1) DEFAULT '保密' COMMENT '员工性别',
    -- 检查约束：年龄必须≥18
    emp_age TINYINT CHECK(emp_age >= 18) COMMENT '员工年龄',
    -- 非空+唯一约束：邮箱不能空、不能重复
    emp_email VARCHAR(50) NOT NULL UNIQUE COMMENT '员工邮箱'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT '员工信息表';
```


#### 问题 3：自增长约束（AUTO_INCREMENT）的使用条件与常见误区是什么？如何修改其初始值？

**答案**：使用条件：① 仅能作用于**整数类型**字段；② 字段必须是 “键列”（主键或唯一约束列）；③ 每表仅能有 1 个自增长约束。常见误区：① 认为自增长值一定从 1 开始（可自定义初始值）；② 给非整数列添加自增长（报错）；③ 插入非零非空值时仍期望自动增长（实际用插入值）。修改初始值的方式：① 建表时指定：`CREATE TABLE 表名(id INT PRIMARY KEY AUTO_INCREMENT) AUTO_INCREMENT=100;`（初始值 100，下一条为 101）；② 建表后修改：`ALTER TABLE 表名 AUTO_INCREMENT=200;`（后续插入值从 200 开始）。
#### 问题 1：主键约束与唯一约束的核心区别是什么？实际开发中如何选择？

**答案**：核心区别体现在 3 点：① 非空要求：主键约束要求 “唯一且非空”，唯一约束仅要求 “唯一”（允许 NULL）；② 数量限制：一个表仅能有 1 个主键，可有多个唯一约束；③ 索引默认：主键约束默认创建主键索引，唯一约束默认创建唯一索引。实际选择：① 需唯一标识行数据（如用户 ID、订单号）→ 用主键约束，推荐自定义主键（如`user_id`）；② 仅需字段值唯一（如手机号、邮箱）→ 用唯一约束（允许 NULL，适配 “未填写” 场景）。

#### 问题 2：外键约束的核心作用与行业实践中的争议是什么？为什么阿里规范不推荐使用？

**答案**：外键约束的核心作用是**保证跨表数据的参照完整性**（如分数表不能引用不存在的学生 ID）。行业争议：① 优点：语法层面强制约束，减少无效数据；② 缺点：限制灵活性，高并发 / 分布式场景下易导致性能瓶颈（级联操作阻塞、插入速度慢）。阿里规范不推荐的原因：① 适配分布式架构：分布式集群中，外键约束难以跨节点维护；② 避免阻塞风险：级联更新 / 删除可能引发数据库更新风暴，导致系统卡顿；③ 简化数据库设计：将数据一致性校验转移到应用层，提高数据库扩展性。
