# MySQL 数据库高级和新特性（2024 企业开发版）核心总结

本次内容围绕 MySQL 高级操作与 8+版本新特性展开，分为**数据库事务、用户权限控制管理、数据备份和还原、MySQL 8+新特性**四大核心模块，以下为各模块关键知识点：

## 一、数据库事务
1. **事务概述**：是不可分割的数据库操作集合，内部命令要么全部执行成功并统一更新数据，要么任一命令失败则回滚；作用为故障恢复、防止并发操作互相干扰，典型场景为转账（减钱+加钱为一个事务）。
2. **ACID 核心特性**
    - 原子性：事务不可分割，操作要么全发生，要么全不发生
    - 一致性：事务前后数据状态一致，避免部分成功产生错误数据
    - 隔离性：事务执行不受其他并发事务干扰
    - 持久性：事务提交后，数据修改永久生效，不受后续操作/故障影响
3. **事务的开启、提交、回滚**：MySQL 默认**自动提交**（单条 SQL 为独立事务），需手动将多条 SQL 纳入一个事务，有两种核心方案，且**DDL 语句（create/drop/alter 等）、TRUNCATE**不支持事务，回滚无效。
    - 方案 1：手动提交模式，`set autocommit = false/0` 关闭自动提交，执行 SQL 后用 `commit` 提交/ `rollback` 回滚，`set autocommit = true/1` 恢复。
    - 方案 2：自动提交下开独立事务，`start transaction` 开启，执行 SQL 后用 `commit/rollback` 收尾。
4. **事务隔离级别**：隔离级别越高，数据一致性越好，并发性能越弱，MySQL 默认**repeatable-read**，可通过 `set transaction_isolation='级别'` 修改、`select @@transaction_isolation;` 查看。

| 隔离级别             | 脏读  | 不可重复读 | 幻读     |
| ---------------- | --- | ----- | ------ |
| read-uncommitted | 是   | 是     | 是      |
| read-committed   | 否   | 是     | 是      |
| repeatable-read  | 否   | 否     | 是（小概率） |
| serializable     | 否   | 否     | 否      |
- 脏读：读取其他事务**未提交**的数据；不可重复读：读取其他事务**提交的修改**数据；幻读：读取其他事务**提交的增删记录**，导致结果集行数变化。

## 二、用户权限控制管理
1. **权限概述**：MySQL 通过权限限制用户对数据库/表的访问，权限可细化到库、表、具体操作级别，核心权限项如下：

| 权限项            | 核心作用         |
| -------------- | ------------ |
| SELECT/INSERT  | 查询 / 插入表数据   |
| UPDATE/DELETE  | 更新 / 删除表数据   |
| CREATE/DROP    | 创建 / 删除库 / 表 |
| GRANT OPTION   | 将自身权限授予其他用户  |
| ALL PRIVILEGES | 拥有所有操作权限     |
2. **核心操作语法**：围绕**创建、赋权、回收、查看、删除用户**展开，`localhost` 仅允许本地连接，`%` 允许任意主机连接。
    - 创建用户：`CREATE USER 'username'@'主机' IDENTIFIED BY 'password';`
    - 赋予权限：`GRANT 权限1,权限2 ON 库.表 TO 'username'@'主机';`（`*` 表示所有）
    - 回收权限：`REVOKE 权限 ON 库.表 FROM 'username'@'主机';`
    - 查看权限：`SHOW GRANTS FOR 'username'@'主机';` / `SELECT User, Host FROM mysql.user;`
    - 删除用户：`DROP USER ` 用户名 `;`

## 三、数据备份和还原
### （一）基础概述
备份是将数据库数据/表结构复制保存，核心应用场景：灾难恢复、数据迁移、数据复制、测试开发环境数据复用。

### （二）全量备份与还原
通过 `mysqldump` 工具实现，**需在未连接 MySQL 的 CMD 中执行**，`-p` 后密码需紧贴参数。
1. 全量备份：`mysqldump -u 用户名 -p 库名 [表1 表2...] > 备份文件.sql`（不写表名则备份全库）
2. 全量还原：`mysql -u 用户名 -p 库名 < 备份文件.sql`（需提前创建目标库，保证 MySQL 版本兼容）

### （三）Binlog 日志恢复误删数据
Binlog 为 MySQL 二进制日志，记录所有增删改查操作及执行时间，可用于误删恢复、主从同步等，默认开启，需在 `my.ini` 中配置 `log-bin=mysql-bin` 指定日志名。
1. 核心日志操作
    - 重置日志：`RESET MASTER;`（清空原有日志）
    - 刷新日志：`FLUSH LOGS;`（生成新日志文件）
    - 查看日志：`SHOW BINARY LOGS;`（日志清单）/ `SHOW BINLOG EVENTS [IN '日志名'];`（日志详情）
    - 解析日志：`mysqlbinlog -v 日志文件`（CMD 中执行，查看日志具体内容）
2. **误删数据恢复思路**：找到删除语句的 `Pos` 起始/终止位置，跳过删除语句，将**删除前、删除后**的日志导出为 SQL 脚本，依次执行脚本恢复；核心工具选项：`--start/stop-position`（按位置筛选）、`--start/stop-datetime`（按时间筛选）。
3. 恢复步骤：导出分段日志脚本→删除原有损坏库→执行脚本恢复（脚本含建库/表语句，无需提前准备）。

## 四、MySQL 8+新特性
本次核心介绍**窗口函数**和**公用表表达式（CTE）**两大核心新特性。

### （一）窗口函数
1. 核心特点：与 `OVER` 子句配合使用，执行聚合/分析/排序操作，**不改变查询结果集的行数**，适用于数据排名、滑动窗口分析。
2. 核心语法：
    - 直接定义：`多行函数 OVER（[PARTITION BY 分组字段 ORDER BY 排序字段 ASC|DESC]）`
    - 定义窗口名复用：`多行函数 OVER 窗口名 ... WINDOW 窗口名 AS（[PARTITION BY 分组字段 ORDER BY 排序字段]）`
3. 函数分类及核心作用（基于商品表场景实现）：
函数分类及核心作用（基于商品表场景实现）：

| 函数分类 | 代表函数                       | 核心作用                                             |
| ---- | -------------------------- | ------------------------------------------------ |
| 序号函数 | ROW_NUMBER/RANK/DENSE_RANK | 生成排序序号，ROW_NUMBER 顺序排、RANK 并列跳号、DENSE_RANK 并列不跳号 |
| 分布函数 | PERCENT_RANK/CUME_DIST     | 计算等级百分比 / 累积分布比例                                 |
| 前后函数 | LAG (expr, n)/LEAD (expr, n)   | 获取当前行的前 n 行 / 后 n 行的指定字段值                        |
| 首尾函数 | FIRST_VALUE/LAST_VALUE     | 获取分区内排序后的第一个 / 最后一个指定字段值                         |
| 其他函数 | NTILE (n)                   | 将分区数据分为 n 个桶，记录桶编号                               |
### （二）公用表表达式（CTE）
1. 核心特点：命名的临时结果集，作用范围为**当前语句**，可复用，支持引用其他 CTE，可替代子查询（子查询无法引用其他子查询）。
2. 分类：普通公用表表达式、递归公用表表达式。
3. 核心语法：`WITH CTE名称 AS（子查询） SELECT/DELETE/UPDATE 语句;`
4. 优势：相比子查询，逻辑更清晰，可多次复用同一临时结果集，适合复杂查询场景。


- 反引号 `` ` ``：仅表名 / 字段名（比如 `order_main`、`user_id`）；
- 英文单引号 `' '`：字符串值（比如 ' 张三 '、' 待支付 '）；
