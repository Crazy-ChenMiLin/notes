# 一.**RedisTemplate 和 Jedis 的连接池的区别** 

| 维度         | JedisPool（Jedis 连接池）                                   | RedisTemplate（Spring 模板）                                                                                                                                                                 |
| ---------- | ------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **定位**     | 底层连接池组件（仅管连接，不管业务操作）                                   | 上层业务操作模板（管连接 + 提供简洁 API + 生态整合）                                                                                                                                                          |
| **所属生态**   | 独立组件（不依赖 Spring），需手动管理                                 | Spring 生态组件，与 Spring 无缝整合（如事务、依赖注入）                                                                                                                                                      |
| **连接管理方式** | 直接管理 Jedis 连接，需手动从池子里借（`getResource()`）、还（`close()`）连接 | 自动管理连接（Spring 帮你借 / 还），无需手动操作连接                                                                                                                                                          |
| **功能范围**   | 仅连接池核心能力（创建、复用、销毁、超时、最大连接数等）                           | 1. 依赖底层连接池（JedisPool/LettucePool）；<br><br>2. 内置序列化（String/JSON/ProtoBuf 等）；<br><br>3. 封装 Redis 所有命令（String/Hash/List/Set 等）；<br><br>4. 支持 Spring 事务、缓存注解（`@Cacheable`）；<br><br>5. 异常统一处理 |
| **使用复杂度**  | 较高：需手动获取 / 释放连接，手动处理异常                                 | 较低：直接调用 API 即可，Spring 接管底层细节                                                                                                                                                             |



# 二. ：Jedis 连接 Redis 的核心步骤有哪些？ 

| 组件                              | 自助餐比喻          | 核心作用                                        |
| ------------------------------- | -------------- | ------------------------------------------- |
| JedisPool                       | 自助餐区           | 存储、复用 Jedis 连接，避免频繁创建销毁                     |
| 你的自定义工具类                        | 取餐窗口           | 帮你从自助餐区拿取餐盘（Jedis 连接）                       |
| Spring 的 JedisConnectionFactory | 自助餐区管理员 + 取餐窗口 | 先搭建自助餐区（创建 JedisPool），       再帮你拿餐盘，还负责餐区维护 |





# 三、RedisTemplate 清底层关系
`
你可以把这个关系类比成：

- RedisTemplate一个 “快递站老板”，负责接收我们的寄件需求，比如 set 或者 get 操作。

- 底层客户端（Lettuce/Jedis）就像是快递员，是实际跑一趟去送快递的人

- Commons-pool 2 就是快递站的 “车辆调度中心”
1. **复用连接**：创建一批连接放在 “池子里”，每次用的时候从池里拿，用完放回，不用重复创建销毁
2. **控制资源**：限制最大连接数（比如配置里的 `max-active:16`），避免连接泛滥
3. **提升性能**：省去连接创建销毁的耗时，让 Redis 操作更快 

- Redis TCP 连接 = 快递车本身（每次和 Redis 通信都需要 1 个 TCP 连接）
