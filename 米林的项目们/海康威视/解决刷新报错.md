# 原因总结：
# 一. SDK 环境
### 以前的逻辑：
引用计数法（当连接数归零时销毁 SDK）
触发场景：浏览器刷新时，所有 WebSocket 连接瞬间断开（计数归零 -> SDK 销毁），紧接着又发起新连接（SDK 初始化）。
	刷新网页时摄像头，系统就把 SDK 彻底关停。SDK 重启过程非常慢且容易卡死。
``` java
if (--sdkInitCount <= 0) { MvCameraControl.MV_CC_Finalize(); } // 致命：并发下误杀 SDK
```


### 现在的逻辑（全局常驻）：
SDK 在程序启动时只初始化一次，SDK 应作为基础设施一直存活，直到 JVM 停止
刷新网页时，仅仅是切断了线路（Close Device），但发电机（SDK 环境）一直在转。所以当您再次打开时，省略了庞大的 SDK 初始化时间，瞬间响应。

MV_CC_Finalize () 是对 SDK 本地运行时做全局清理（释放线程、socket、内存、内部缓存）。它不是单个设备关闭的代替品。
在多连接/并发场景，以连接数为准去 Finalize 会产生竞态条件（race condition）：短时间内既有销毁又有重建请求，容易出错。


# 为什么把 SDK Finalize 移除/改为全局常驻能解决问题？
因此更稳健的策略是：
在每次客户端断开时，只执行设备级的释放：
	MV_CC_StopGrabbing (h)
	MV_CC_CloseDevice (h)
	MV_CC_DestroyHandle (h)
仅在应用完全退出时（例如 JVM 停止、服务进程关闭）再调用全局 MV_CC_Finalize ()。
